# KW-OKF Profile Schema

Use Markdown files with a YAML-compatible frontmatter block. The glue script writes JSON-compatible scalar/list/object values inside frontmatter so it can parse them without external YAML dependencies.

KW-OKF is an opinionated producer profile inspired by the Open Knowledge Format and Karpathy's LLM-wiki pattern. Its internal `ROOT_INDEX` / `ROUTER` / `LEAF_RULE` model, Router frontmatter, audit fields, and staged-write metadata are KW-OKF extensions; the internal Vault does not claim direct OKF v0.1 conformance. Use `export-okf` to generate a handoff bundle with reserved frontmatter-free `index.md` files and business-semantic concept `type` values.

## Node Types

Inside a KW-OKF Vault, `type` is structural only:

| type | path pattern | parent rule |
| --- | --- | --- |
| `ROOT_INDEX` | `index.md` | no parent required |
| `ROUTER` | `wiki/<fixed-category>/<topic>/index.md` | parent is `ROOT_INDEX` or another `ROUTER` |
| `LEAF_RULE` | `wiki/<fixed-category>/<topic>/<slug>.md` | parent must be a `ROUTER` |

Business semantics go in `knowledge_type`, not `type`. Business Router directories and note filenames are human-readable semantic names chosen by Codex from the current task context; they may be Chinese, English, or mixed. Non-ASCII paths generate ids from an ASCII prefix plus a short UTF-8 SHA1 hash to prevent collisions.

## LEAF_RULE Frontmatter

Required fields: `id`, `type`, `parent_id`, `parent_path`, `knowledge_type`, `status`, `title`, `summary`, `aliases`, `tags`, `scope`, `confidence`, `source_refs`, `images`, `links`, `supersedes`, `superseded_by`, `created_at`, `updated_at`, `review_after`, `language`.

Audit checks `ROUTER` / `LEAF_RULE` path shape, path-derived `id`, parent existence, `parent_id` equality with the parent page frontmatter `id`, required fields and list field types, timestamps, body structure, assets, and links.

## Field Semantics and Write Rules

Keep frontmatter keys, JSON keys, CLI flags, and enum values in English. Human-readable field values such as `title`, `summary`, `aliases`, `tags`, and `scope` may be Chinese, English, or mixed when semantically clear.

| Field | Purpose | Write rule |
| --- | --- | --- |
| `id` | Unique machine identity of the current page | Generated by the script from the Vault-relative path and required to match that path; keep it as a stable ASCII key. Humans may inspect it for debugging, but do not edit it for readability. |
| `type` | Structural node type | Use only the fixed enum; do not put business semantics here. |
| `parent_id` | Machine ID of the parent node | Required for `ROUTER` and `LEAF_RULE`; must equal the parent page frontmatter `id`. `ROOT_INDEX` does not need it. |
| `parent_path` | Vault-relative path of the parent node | Helps humans confirm hierarchy and lets audit check parent-child consistency; must point to the parent page. |
| `knowledge_type` | Business semantic type | Use only the fixed enum such as rule, procedure, decision, or case; do not mix it with `type`. |
| `status` | Page lifecycle state | Use `active` for normal committed knowledge; use `deprecated` or `superseded` only when the knowledge is obsolete or replaced. |
| `title` | Human-readable title | Follow the current user language or explicit user language; keep it short, clear, and stable. |
| `summary` | Page summary | Explain in 1-2 sentences what problem this page solves; follow the body language. |
| `aliases` | Alternative names and search synonyms | May be Chinese, English, or mixed; use them for recall, but do not stuff unrelated keywords. |
| `tags` | Retrieval facets | May be Chinese, English, or mixed; prefer existing tags from root `tags.md`; new tags must have clear meaning and be shown in the preview, and the same concept should stay stable unless cross-language recall is intentional. |
| `scope` | Applicability boundary | State the project, scenario, or boundary where this knowledge applies; follow the body language. |
| `confidence` | Evidence confidence | Use only the fixed enum; choose it from evidence strength and do not inflate certainty. |
| `source_refs` | Evidence sources | Record conversation, file, URL, or manual-note sources; this is not a body summary and must help trace evidence. |
| `images` | Image or asset references | Include only Vault-relative asset paths that support this page. |
| `links` | Cross-note associations | Use standard Markdown relative links; dangling links are allowed as warnings, but do not use Obsidian `[[WikiLinks]]`. |
| `supersedes` | Older knowledge replaced by this page | Fill only when this page replaces older knowledge; prefer the replaced page `id` or Vault-relative path. |
| `superseded_by` | Newer knowledge replacing this page | Fill only when this page has been replaced; prefer the new page `id` or Vault-relative path. |
| `created_at` | Creation timestamp | ISO 8601 timestamp written by the script. |
| `updated_at` | Update timestamp | ISO 8601 timestamp; update when the content changes materially. |
| `review_after` | Review date | Reminder for future review; useful for knowledge likely to expire. |
| `language` | Body language | Supports configured languages. Resolve explicit user choice first, otherwise follow the current conversation language, then use `fallback_language` only when unclear. |

Valid values:

- `type`: `ROOT_INDEX`, `ROUTER`, `LEAF_RULE`
- `knowledge_type`: `RULE`, `SOP`, `DECISION`, `CASE`, `ENTITY`, `SOURCE`, `VISUAL_ASSET`
- `status`: `active`, `draft`, `deprecated`, `superseded`
- `confidence`: `low`, `medium`, `high`
- `language`: `en-US`, `zh-CN`

## Tag Registry

Root `tags.md` records the global tag registry. It does not replace the page frontmatter `tags` field. Page `tags` store tag strings only; tag meanings, applicability boundaries, aliases, and discouraged spellings belong in `tags.md`.

Recommended structure:

| Tag | Meaning | Applicability | Aliases / discouraged spellings | Status | Example |
| --- | --- | --- | --- | --- | --- |
| `kw-okf-memory` | KW-OKF long-term memory Skill and Vault topics | Use only for this long-term memory system | `okf-memory`, `long-term-memory` | `active` | `wiki/projects/kw-okf-memory/...` |

Recommended tag statuses: `active`, `deprecated`, `merged`. Before adding a tag, check `tags.md` and `categories.json`; if a synonym already exists, prefer reusing it or propose a merge in the preview.

## Source Refs

`source_refs` is an array of objects. Common fields: `type`, `id`, `path`, `timestamp`, `excerpt`, `confidence`.

Path validation:

- `url`: `path` must start with `http://` or `https://`.
- `file`: `path` may be a local file path and does not need to be inside the Vault.
- `vault_note`: `path` must be a Vault-relative path.
- `codex_thread`, `chat_excerpt`, `manual_note`: `path` may be empty, but `id` or `excerpt` must exist.

## Body Template

Use the English template when the resolved note language is `en-US`:

```markdown
# Title

## One-Sentence Conclusion

## Scope

## Rules

## Evidence and Sources

## Counterexamples and Risks

## Related Knowledge

## Evolution Log
- 2026-07-08 ADD: ...
```

Use the Chinese template when the resolved note language is `zh-CN`:

```markdown
# 标题

## 一句话结论

## 适用边界

## 操作规则

## 证据与来源

## 反例与风险

## 关联知识

## 知识演进日志
- 2026-07-08 ADD: ...
```

The glue script treats the one-sentence conclusion, scope, evidence, and evolution-log headings as the minimum preferred structure. Different knowledge types may add sections, but rules and SOPs must retain a clear applicability boundary and evolution history.

## Staging Metadata

Staged drafts include `_kw_target_path` to bind the previewed draft to its intended formal path. `commit` rejects a different target and removes this staging-only key before writing the formal page.

## Official OKF Export

`export-okf` maps each `LEAF_RULE` to an OKF-style concept document, maps `knowledge_type` to the exported concept `type`, preserves KW-OKF metadata under `kw_*` extension keys, and generates reserved `index.md` / `log.md` files without Router frontmatter. The export is a derived handoff artifact; the KW-OKF Vault remains the truth source.
